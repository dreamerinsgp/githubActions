### 二、测试策略

| 问题 | 回答 |
|------|------|
| 交易服务测试重点？ | 场景拆分（按交易类型）→ 流程梳理 → 正常case → 边界case → 非场景（网络、设备）→ 性能压测 → 回归自动化（GitHub Actions）。 |
| 保证质量的最重要测试点？ | 1. 可用性：链路功能可用 + 高并发下系统稳定。2. 安全性：交易前鉴权、交易中全链路监控、交易后落库；防恶意攻击、主从备份防单点。 |

### 三、系统架构

| 问题 | 回答 |
|------|------|
| 交易服务系统架构？ | APP → Gateway(HTTP→RPC) → trade RPC。鉴权在网关层。trade内：调用helios查余额 → 交易构造（从DB取池子账户）→ 签名服务(TEE) → RPC发送（Jito私有/公开节点）→ 链共识。返回用confirmed级别。 |
| 防夹措施？ | Jito私有节点，无公开mempool，夹子检测不到。 |
| 落库同步还是异步？ | 主从备份，理解是异步。 |

### 四、性能测试

| 问题 | 回答 |
|------|------|
| 如何评估承载能力？ | 单接口压测，先定基准（低线程下的TPS、CPU/内存/磁盘），再调线程数找瓶颈。基准：CPU<80%，内存<70–80%。 |
| 压测数据？ | 市价单约3000+/s（无外部依赖），限价单约2000+/s（涉及MySQL、Redis）。 |
| 测试与线上配置一比一？ | 单台配置相同，数量不同（测试1台，线上多台）。数据量尽量一致。 |
| 单接口压测？ | 是，4–5个核心接口分别压。 |
| 分析过线上流量？ | 有分析，但用户量不大，做得少。 |
| 热门币对（如必然人生）？ | 川普币时量较大，做过扩容准备。 |
| 混合流量场景压测？ | 做得少，主要单接口摸底。 |
| 用测试链？ | 是，不用主网。 |
| 瓶颈在哪？ | 主要在MySQL，CPU约60%，慢查询。做了SQL优化缓解。 |
| 提升容量的方式？ | 硬件扩容（数据库、更好配置），第三方RPC限速（helios 999美金/月约50次/s），考虑自建节点。 |
| 异步落库为何DB仍是瓶颈？ | 面试官指出：异步落库时DB不应阻塞主链路，需细节分析。 |
| 压测观察哪些指标？ | CPU、内存、Redis、MySQL占用率，磁盘读写能力及趋势。 |
